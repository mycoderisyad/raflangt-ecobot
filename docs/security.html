Security Measures

Webhook Authentication

  • Both WhatsApp and Telegram webhooks use HMAC-based secret verification
  • Fail-closed design: if the secret env var is not configured, all requests are rejected
  • Uses hmac.compare_digest() for constant-time comparison (prevents timing attacks)

Admin Panel Security

  • CSRF protection via flask-wtf CSRFProtect on all POST forms
  • Rate limiting on login: 5 requests/minute, 20 requests/hour
  • Session hardening: HttpOnly, SameSite=Strict, Secure (production), 8-hour timeout
  • Constant-time credential comparison with hmac.compare_digest()

API Security

  • User management endpoints require X-API-Key header
  • Key validation uses constant-time comparison
  • Returns 500 if API_SECRET_KEY is not configured (fail-closed)

SSRF Protection

  • Media download validates URL scheme (only http/https)
  • Resolves hostname and blocks RFC1918, loopback, and link-local IP ranges
  • Enforces 5 MB max download via Content-Length and streaming

Infrastructure

  • Docker runs as non-root appuser
  • PostgreSQL bound to 127.0.0.1 only in Docker Compose
  • Startup validation rejects weak/placeholder secrets in production

Production Recommendations

  1. Set strong, unique values for WAHA_WEBHOOK_SECRET, TELEGRAM_WEBHOOK_SECRET, API_SECRET_KEY, ADMIN_PANEL_SECRET_KEY, and ADMIN_PANEL_PASSWORD
  2. Enable HTTPS via reverse proxy (Nginx/Caddy) so SESSION_COOKIE_SECURE works
  3. Use Redis-backed flask-limiter for multi-worker deployments
  4. Rotate all secrets if they were ever committed to version control
  5. Run security PoC tests after any webhook or auth-related code changes
  6. Keep Python dependencies updated (pip audit)
